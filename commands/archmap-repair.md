# /archmap:repair — Full Architecture Map Repair

Detect and fix staleness, layout issues, missing details, and template corruption in existing architecture maps. Does NOT regenerate from scratch — surgically patches what's broken.

## Usage

```
/archmap:repair           # Smart repair — detect and fix all issues
/archmap:repair --layout  # Fix only layout problems (overlaps, spacing)
/archmap:repair --details # Fix only missing module details (types, functions, notes)
/archmap:repair --full    # Force re-validate everything including template integrity
```

## Prerequisites

An existing `docs/architecture.html` must exist (generated by `/archmap`). If no map exists, tell the user to run `/archmap` first.

## Behavior

### Phase 0: Auto-snapshot

Before mutating anything, preserve the pre-repair map by calling `/archmap:snapshot` with `--note "auto: pre-repair"` unless either of these is true:

- The user passed `--no-snapshot` on this invocation
- `.archmap.json` has `"history": { "autoSnapshotOnRepair": false }`

This guarantees users can always roll back a bad repair by scrubbing to the pre-repair snapshot in the Phase 2 timeline UI.

### Phase 1: Extract Current Map State

1. **Read** `docs/architecture.html` from the project root (or custom path from `.archmap.json`)
2. **Parse** the embedded JavaScript to extract the current map data:
   - Find the line `const modules = [...]` and extract the JSON array
   - Find `const edges = [...]` and extract the JSON array
   - Find `const tierLabels = [...]` and extract
   - Find `const pipelineSteps = [...]` and extract
   - Find `const legendItems = [...]` and extract
   - Find `const layoutOverrides = {...}` and extract
   - Find `const history = [...]` and extract (preserve across the rewrite)
   - Find the project name from the `<title>` tag
3. **Store** this as the "current map state" for diffing

### Phase 2: Targeted Re-exploration

1. **Check** for `.archmap.json` in the project root — if present, load exclude paths and tier overrides
2. **Dispatch** `archmap:archmap-repair-agent` in **scan mode** via the Task tool:
   ```
   subagent_type: archmap:archmap-repair-agent
   ```
   Provide the agent with:
   - The full list of current module objects (id, label, tier, file paths from details.imports context)
   - The project root path
   - The exclude paths from `.archmap.json` (if present)
3. **Wait** for the repair report

### Phase 3: Analyze & Fix

Process the repair report and apply fixes to each category:

#### Staleness Fixes
- **Dead modules:** Remove from modules array, remove all edges referencing them, remove from legend if their tier is no longer used
- **Stale modules:** Update their details (types, functions, imports, line count) with the new data from the report
- **New modules:** Add to modules array with appropriate tier, position, and details. Position them in the correct tier column with proper spacing.
- **Broken edges:** Remove from edges array
- **New edges:** Add to edges array with descriptive labels

#### Layout Fixes
After modules are added/removed, recalculate layout:
- **Tier columns:** Group modules by tier, arrange in columns left-to-right
- **Vertical spacing:** Stack modules within each column with ~20px gaps
- **Column spacing:** ~240px between tier columns
- **Box sizing:** Width 130-170px, height 50-80px based on line count
- **Tier labels:** Recalculate `tierLabels` positions above each column
- **Orphan detection:** If a module has zero edges, flag it (keep it but position it in a "misc" area)

If `--layout` flag is used, ONLY perform layout fixes (skip staleness/details).

#### Detail Fixes
For modules flagged with detail gaps:
- If types are empty but the module has source files, the repair agent should have found them
- If functions are empty, same
- If notes are empty, generate a brief architectural note based on the module's role, imports, and tier
- Update `details.imports` to match current edges

If `--details` flag is used, ONLY perform detail fixes (skip staleness/layout).

#### Template Integrity (only with --full)
- Validate the HTML structure is well-formed
- Check that all JSON data parses correctly
- Verify all tier values used in modules exist in the THEMES objects
- Check that module IDs in edges reference existing modules

### Phase 4: Patch & Rewrite

1. **Read** the template from `${CLAUDE_PLUGIN_ROOT}/templates/archmap-template.html`
2. **Replace** placeholders with the repaired data:
   - `{{PROJECT_NAME}}` → project name (preserved from existing map)
   - `{{STATS_HTML}}` → regenerated stats from repaired module data
   - `{{MODULES_JSON}}` → repaired modules array
   - `{{EDGES_JSON}}` → repaired edges array
   - `{{TIER_LABELS_JSON}}` → recalculated tier labels
   - `{{PIPELINE_JSON}}` → updated pipeline (if data flow changed)
   - `{{LEGEND_JSON}}` → updated legend (only tiers actually used)
   - `{{LAYOUT_JSON}}` → merged user-arranged positions (see below)
   - `{{HISTORY_JSON}}` → the existing `history` array (preserved verbatim from Phase 1, now including the pre-repair auto-snapshot added in Phase 0). Never discard history.
3. **Preserve user layout.** Extract the existing map's `const layoutOverrides = {...}` block AND the `modules[].x / .y` positions — they represent user-arranged positions. Merge those into the new `{{LAYOUT_JSON}}` so repair never loses manually-placed modules. For each module in the repaired result:
   - If it existed in the old map: keep its old x/y in `layoutOverrides`.
   - If it's newly added: auto-layout in its tier column (do NOT add to `layoutOverrides` — let defaults stand so the user can decide where it belongs).
   - If the user's manual position would now collide with a new module, flag the collision in Phase 5 so the user can rearrange if they want.
   Also merge `.archmap/layout.json` (if present) into the result — it's the source of truth for shared layouts.
4. **Write** to `docs/architecture.html` (or custom path from `.archmap.json`)
5. **Regenerate** `docs/architecture-map.md` from the repaired data using the same markdown format as `/archmap` Phase 3.5
6. **Open** in browser

### Phase 5: Report

Tell the user:
- How many issues were detected in each category
- What was fixed (added/removed/updated modules, edges, layout changes)
- Any modules that need manual review (ambiguous tier assignments, uncertain merges)
- Remind them to review the repaired map and commit if satisfied

## Important

- Never discard user-customized notes or descriptions — preserve `details.notes` from existing modules unless the module was deleted
- If `.archmap.json` has `pinned` modules, never remove or re-tier them even if the repair agent suggests changes
- The pipeline sidebar should be updated if the data flow fundamentally changed (new entry points, removed outputs)
- Module `color` properties are set at runtime by `applyTheme()` — do NOT hardcode colors
- Every `tier` value used MUST exist in the THEMES.*.tiers objects
